Regras obrigatórias:

- não executar drop/reset/truncate;
- fazer apenas alterações incrementais;
- manter campanhas, contatos, integrações, relatórios e atendimentos existentes;
- manter SMS via proxy n8n funcionando;

  npm run check

Correções necessárias:

1. Remover aviso de templates em canal não oficial
Na etapa Mensagem, quando o canal selecionado for WhatsApp normal/não oficial, SMS ou E-mail, não deve aparecer o aviso:
“Templates oficiais disponíveis apenas para WhatsApp API Oficial...”

Esse aviso está poluindo a tela para canal não oficial.

Regra correta:
- se o canal/número selecionado NÃO for WhatsApp API Oficial, esconder completamente a seção/aviso de templates oficiais;
- mostrar apenas campo de mensagem livre;
- só mostrar templates oficiais quando o número/conexão selecionada for realmente da API Oficial.

2. Unificar WhatsApp normal e WhatsApp API Oficial
Hoje o wizard mostra “WhatsApp” e “WhatsApp API Oficial” como canais separados. Isso está confuso.

Corrigir para:
- ter apenas um canal chamado “WhatsApp”;
- na configuração/etapa de canal, o usuário deve escolher o número/conexão de envio;
- cada número/conexão deve indicar se é:
  - WhatsApp normal;
  - WhatsApp API Oficial.

Fluxo esperado:
- usuário escolhe canal “WhatsApp”;
- depois escolhe “Número de envio”;
- se o número escolhido for WhatsApp normal:
  - permitir mensagem livre;
  - não carregar templates oficiais;
- se o número escolhido for WhatsApp API Oficial:
  - listar templates aprovados;
  - permitir selecionar template;
  - permitir preencher variáveis do template;
  - permitir disparar usando template oficial.

Remover a opção separada “WhatsApp API Oficial” da lista principal de canais.

3. Corrigir erro SMS Oktor 502
Ao disparar SMS, apareceu erro:
“Oktor SMS 502: <html>... 502 Bad Gateway ... nginx ...”

Esse erro provavelmente veio do proxy n8n ou da Oktor, mas o sistema está mostrando HTML bruto para o usuário.

Corrigir:
- tratar respostas 502/503/504 do SMS/proxy como erro amigável;
- não exibir HTML bruto na interface;
- mostrar mensagem amigável:
  “Serviço de SMS temporariamente indisponível. Tente novamente em alguns minutos.”
- registrar o erro técnico completo apenas no log/auditoria;
- manter o destinatário como falha temporária ou pendente de retry, não como erro definitivo se houver retry configurado;
- se possível, aplicar retry automático para 502/503/504.

Também validar:
- se o endpoint usado é o proxy n8n e não Oktor direto;
- log seguro: não expor token/code/account no frontend.

4. Status de entrega/leitura do WhatsApp em campanhas
Na página de detalhe da campanha, as mensagens WhatsApp aparecem apenas como “Enviado”, mas não mostra se foi entregue/lido.

Corrigir:
- capturar/atualizar status de mensagens WhatsApp quando houver webhook/evento de entrega/leitura;
- vincular status da mensagem ao destinatário da campanha;
- atualizar campos:
  - enviado em;
  - entregue em;
  - lido em;
  - respondido em, se aplicável;
  - erro, se houver;
- na tabela de destinatários da campanha, mostrar status real:
  Enviado, Entregue, Lido, Respondido, Falhou, Pendente;
- atualizar métricas do dashboard da campanha com esses status.

Se o provedor/conexão não fornecer entrega/leitura, mostrar:
“Status de entrega indisponível para este canal”
em vez de deixar parecer erro.

5. Variáveis não funcionam em contatos manuais/listas
As variáveis como {nome}, {telefone}, {cidade}, {protocolo}, {link} não estão funcionando corretamente quando os contatos são adicionados manualmente ou por lista/importação.

Corrigir:
- no modo manual, permitir informar nome junto com telefone.
Exemplos aceitos:
  +5551999462387;Guilherme
  +5551999462387,Guilherme
  Guilherme;+5551999462387
  Guilherme,+5551999462387

- no modo importar/colar lista, mapear colunas como:
  nome;telefone;email;cidade
  João;51999999999;joao@email.com;Chapecó

- garantir que cada destinatário mantenha os campos:
  nome
  telefone
  email
  cidade
  estado
  etiquetas
  campos personalizados, se existirem

- quando enviar mensagem:
  {nome} deve virar o nome daquele contato;
  {telefone} deve virar o telefone;
  {cidade} deve virar cidade;
  {protocolo} deve gerar ou usar protocolo da campanha/atendimento;
  {link} deve usar o link informado/configurado ou ficar vazio de forma segura.

- no preview da mensagem, usar um contato real da audiência quando existir;
- se não houver nome, usar fallback seguro, sem quebrar a mensagem.

Corrigir:

  Brenda

O sistema não deve considerar Brenda e Gui como números separados.
Deve interpretar como pares telefone/nome quando possível, ou orientar o usuário a usar:
  +5551981685364;Brenda
  +5551999462387;Gui

Melhorar UX:
